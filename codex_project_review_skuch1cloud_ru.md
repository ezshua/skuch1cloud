# Ревью проекта skuch1cloud от Codex

**Проект:** Telegram-бот для хранения и возврата файлов пользователей  
**Стек:** Python 3.10+, aiogram 3.x, aiohttp, python-dotenv  
**Дата ревью:** 2026-05-27  
**Охват:** текущее состояние репозитория, с учетом сравнения с `code_review_skuch1cloud.md`

## Краткий вывод

Существующее ревью в целом полезное и правильно указывает направление, но часть замечаний уже устарела, а часть имеет завышенный или заниженный приоритет. Проект небольшой и понятный, но самые рискованные зоны сейчас - целостность данных, корректность установки/упаковки и блокирующий файловый ввод-вывод внутри async-обработчиков.

Проверка синтаксиса прошла успешно:

```powershell
python -m compileall -q src
```

Мой рекомендуемый порядок работ:

1. Защитить изменения `files_data.json` от конкурентных записей.
2. Исправить повторное выбрасывание исключений и коллизии временных файлов при URL-загрузках.
3. Исправить packaging и явно объявить прямые зависимости.
4. Вынести тяжелые файловые операции из event loop.
5. Разделить обработчики на routers/services после исправления проблем корректности.
6. Добавить точечные тесты на имена файлов и файловый индекс.

## С чем я согласен в существующем ревью

### Race condition вокруг `files_data.json`

Согласен: это одна из главных проблем корректности.

`save_incoming_file()` читает текущий индекс, генерирует отображаемое имя, скачивает файл, а затем добавляет запись в `files_data.json`. При одновременных загрузках одного пользователя несколько обработчиков могут прочитать одно и то же старое состояние, а потом записать дубли или потерять часть обновлений.

Связанный код:

- `src/file_handler.py`: `files_data = get_user_files(destination_dir)`
- `src/file_handler.py`: `append_file_data(destination_dir / "files_data.json", file_info)`
- `src/utils.py`: `append_file_data()` делает read-modify-write без lock

`unique_path()` защищает физический файл от перезаписи, но не делает безопасным обновление JSON-индекса.

### `raise e` нужно заменить на голый `raise`

Согласен. В `file_handler.py` и `url_handler.py` после cleanup используется `raise e`. Это ухудшает traceback и затрудняет отладку.

Нужно заменить на:

```python
raise
```

### Блокирующий файловый I/O в async-коде

Согласен, но с нюансом. Не каждый `Path.exists()` нужно срочно выносить в отдельный поток. Но чтение/запись JSON и особенно рекурсивный обход директорий могут блокировать event loop.

Наиболее заметные места:

- `get_dir_size()` рекурсивно обходит директории.
- `/status` синхронно считает общий размер хранилища.
- проверки квоты при Telegram- и URL-загрузках вызывают `get_dir_size()`.
- admin-отчеты читают JSON-файлы каждого пользователя и считают размеры директорий.

Связанный код:

- `src/utils.py`: `get_dir_size()`
- `src/handlers.py`: `_get_status_content()`
- `src/file_handler.py`: `save_incoming_file()`
- `src/url_handler.py`: `download_file_from_url()`
- `src/utils.py`: `collect_daily_report()`, `collect_users_summary()`

### `handlers.py` слишком большой

Согласен, но это не первый фикс. Сейчас файл содержит создание dispatcher, in-memory состояние, форматирование сообщений, пагинацию статуса, подтверждение удаления, загрузку файлов, URL-логику, поиск файлов и admin-отчеты.

Это ухудшает тестируемость и поддержку. Но если сначала разнести файл по модулям, не исправив общую работу с индексом файлов, та же проблема просто окажется размазана по нескольким файлам.

### Ротация лога слабая

Согласен. `log_user_action()` загружает весь JSON-лог, после превышения лимита удаляет только около 2% записей, добавляет новую запись и полностью перезаписывает файл.

Файл может долго оставаться выше лимита, а каждый новый log write будет снова делать дорогую O(N)-операцию.

Связанный код:

- `src/utils.py`: `log_user_action()`

### Отчеты имеют N+1 filesystem pattern

Согласен. `collect_daily_report()` и `collect_users_summary()` читают файлы каждого пользователя и считают размеры директорий на лету. Для маленького личного бота это приемлемо, но по мере роста числа пользователей и файлов станет заметно медленным.

Связанный код:

- `src/utils.py`: `collect_daily_report()`
- `src/utils.py`: `collect_users_summary()`

### Тестов нет

Согласен. Самые полезные первые тесты - не большие Telegram integration tests, а маленькие unit-тесты вокруг нормализации имен, сокращения отображаемых имен, обновления индекса, дублей и удаления.

## С чем я не согласен или что уточнил бы

### Замечание про `texts.py` устарело

В существующем ревью сказано, что `get_welcome_message()` может отсутствовать в `texts.py` или случайно находиться в `url_handler.py`. В текущем коде это неверно.

Текущее состояние:

- `src/texts.py` определяет `get_welcome_message()`.
- `src/handlers.py` импортирует его из `texts`.
- `src/url_handler.py` не определяет `get_welcome_message()`.

По этому конкретному пункту ничего исправлять не нужно.

### FSM сам по себе не дает persistent state

В исходном ревью предлагается заменить `pending_deletions` и `user_status_msgs` на aiogram FSM. Сама идея нормальная, но приоритет и формулировку нужно уточнить.

`MemoryStorage` все равно потеряет состояние подтверждения удаления после рестарта. Если нужна устойчивость к рестартам, нужен `RedisStorage`, другой persistent backend или собственное persisted-хранилище pending operations.

Кроме того, `user_status_msgs` - это в основном UI-состояние для удаления старого сообщения статуса. Потеря этого состояния при рестарте не критична. Значимое состояние - `pending_deletions`.

### `BOT_DATA_PATH` как path traversal - завышенный риск

`BOT_DATA_PATH` приходит из конфигурации окружения, а не из пользовательского ввода Telegram. Если атакующий может менять `.env`, он уже контролирует конфигурацию процесса.

Проверку или ограничение директории данных можно добавить для защиты от ошибки оператора, но я бы не классифицировал это как критическую security-уязвимость.

### URL filename traversal в основном закрыт

URL handler использует `Path(parsed_url.path).name`, а потом `normalize_filename()`, которая убирает разделители путей и небезопасные символы. Прямой path traversal через имя файла в текущем flow маловероятен.

Более сильный URL-риск - SSRF: бот принимает произвольные HTTP/HTTPS ссылки и делает запросы с серверной машины.

## Дополнительные замечания

### Прямая зависимость от `aiohttp` не объявлена

`src/url_handler.py` напрямую импортирует `aiohttp`, но `aiohttp` не указан ни в `pyproject.toml`, ни в `requirements.txt`.

Даже если сейчас `aiohttp` приходит транзитивно через aiogram, прямой импорт должен быть прямой зависимостью. Иначе обновления зависимостей могут неожиданно сломать запуск.

Рекомендация:

- добавить `aiohttp` в `pyproject.toml`;
- добавить `aiohttp>=...` в `requirements.txt`.

### Packaging настроен, вероятно, неправильно

В `pyproject.toml` указано:

```toml
[tool.setuptools]
package-dir = { "" = "src" }
py-modules = ["main"]
```

Но проект использует несколько top-level модулей из `src`: `bot`, `handlers`, `utils`, `config`, `users`, `file_handler`, `url_handler`, `texts`.

Из-за этого `pip install -e .` может установить проект неполно или зависеть от текущей рабочей директории.

Рекомендуемые варианты:

1. Перенести код в настоящий пакет, например `src/skuch1cloud/...`, и использовать package imports.
2. Или явно перечислить все top-level modules в `py-modules`.

Пакетный вариант лучше для долгосрочной поддержки.

### URL-загрузки могут конфликтовать по временным файлам

`download_file_from_url()` пишет во временный файл:

```python
tmp_path = destination_dir / f"{final_name}.download"
```

Две одновременные загрузки, которые дают одно и то же имя, могут писать в один и тот же временный файл. Для Telegram-загрузок сделано безопаснее: там временное имя использует `content.file_id`.

Рекомендация:

- использовать уникальное временное имя, например через `uuid4().hex`;
- сохранить текущую логику `unique_path()` для финального файла.

### SSRF-риск в URL-загрузках

Бот скачивает произвольные HTTP/HTTPS URL. В облачной среде это может позволить делать запросы к внутренним адресам, localhost-сервисам или metadata endpoints.

Рекомендации:

- резолвить hostname и блокировать private, loopback, link-local, multicast и reserved IP ranges;
- проверять не только исходный URL, но и redirect targets;
- ограничить количество redirect'ов.

### Парсинг `Content-Disposition` хрупкий

Текущий regex - практичный shortcut, но не полноценный parser для RFC 6266 / RFC 5987. Он может некорректно обработать quoted semicolons, необычные кодировки и дублирующиеся параметры.

Это не срочная проблема, но если URL-загрузки станут важной частью продукта, лучше использовать более структурированный parser или оставить fallback максимально консервативным.

### Можно почистить импорты

Есть неиспользуемые или вводящие в заблуждение импорты:

- `append_file_data` импортируется в `handlers.py`, но не используется;
- `slugify_cyrillic_to_ascii` импортируется в `file_handler.py`, но напрямую не используется.

Риск низкий, но при рефакторинге это стоит почистить.

### `ADMIN_ID = 0` как sentinel работает, но выражает намерение хуже

Практической проблемы с Telegram user ID `0` почти нет, но `None` лучше выражает отсутствие администратора.

Рекомендуемый вариант:

```python
_admin_raw = os.getenv("ADMIN_ACCOUNT_ID")
ADMIN_ID: int | None = int(_admin_raw) if _admin_raw else None
```

После этого проверки и command filters должны явно учитывать `None`.

## План реализации

### Этап 1: исправления корректности

Цель: предотвратить порчу данных и улучшить диагностику без большого архитектурного переписывания.

1. Добавить per-user или per-index `asyncio.Lock` для `files_data.json`.
2. Создать helper-функции для атомарных операций с индексом:
   - загрузить индекс;
   - добавить запись о файле;
   - удалить запись о файле;
   - обновить индекс после scan/fix.
3. Использовать эти helper-функции в:
   - сохранении Telegram-файлов;
   - сохранении URL-файлов;
   - подтверждении удаления;
   - scan/fix.
4. Заменить `raise e` на `raise`.
5. Использовать уникальные временные имена для URL-загрузок.
6. Добавить `aiohttp` как прямую зависимость.

### Этап 2: packaging и deployment cleanup

Цель: сделать установку предсказуемой.

1. Перенести модули `src/*.py` в пакет `src/skuch1cloud/*.py`.
2. Обновить импорты на package-relative или absolute package imports.
3. Обновить `pyproject.toml`, чтобы setuptools находил пакет.
4. Добавить console script entry point, например:

```toml
[project.scripts]
skuch1cloud = "skuch1cloud.main:main"
```

5. Обновить README: запускать установленный entry point или `python -m skuch1cloud`.

### Этап 3: async performance improvements

Цель: перестать блокировать event loop тяжелыми файловыми операциями.

1. Обернуть рекурсивные расчеты размера директорий через `asyncio.to_thread()` в async handlers.
2. Рассмотреть кэширование общего размера хранилища и инкрементальное обновление после save/delete.
3. Вынести admin-отчеты в thread/background execution, если они останутся файлово-тяжелыми.

### Этап 4: структура handlers

Цель: улучшить поддержку после стабилизации общего поведения с данными.

Рекомендуемое разбиение:

- `handlers/commands.py`: `/start`, `/status`;
- `handlers/delete.py`: `/delete` и confirmation flow;
- `handlers/files.py`: входящие Telegram media и поиск/возврат файла;
- `handlers/url.py`: обработка URL-сообщений;
- `handlers/admin.py`: `/report`;
- `services/file_index.py`: JSON index operations и locks;
- `services/storage.py`: quota и directory-size logic;
- `formatters.py`: helpers для сообщений.

Для aiogram стоит использовать отдельный `Router` на каждый handler module и подключать routers при создании dispatcher.

### Этап 5: persistent state

Цель: выбрать правильную модель состояния для подтверждения удаления.

Варианты:

1. Оставить in-memory state, если это single-process личный бот и потеря состояния при рестарте приемлема.
2. Использовать aiogram FSM с `MemoryStorage` ради более чистой структуры handlers, но без restart persistence.
3. Использовать aiogram FSM с `RedisStorage`, если нужна устойчивость к рестартам или multi-process режим.
4. Использовать небольшой JSON/SQLite pending-operation store, если Redis слишком тяжел для проекта.

Моя рекомендация: не вводить Redis, если он уже не нужен инфраструктурно. Для размера этого проекта JSON/SQLite-backed service может быть проще и понятнее.

### Этап 6: тесты

Цель: покрыть поведение, которое вероятнее всего сломается при рефакторинге.

Начать с unit-тестов для:

- `normalize_filename()`;
- `slugify_cyrillic_to_ascii()`;
- `shorten_name()`;
- `_clean_filename()`;
- append/remove записей индекса при дублях;
- уникальности временного пути URL-загрузок;
- удаления записи из индекса.

После этого добавить небольшие integration-style тесты вокруг service-функций, без реальных Telegram network calls.

## Финальная оценка

Проект рабочий и достаточно простой для развития, но сейчас он использует JSON-файлы как маленькую базу данных без достаточной защиты от конкурентных изменений. Это нужно исправить первым делом.

Я бы не начинал с полного переписывания на FSM, Redis или SQLite. Практичный путь: сначала стабилизировать файловый индекс и установку проекта, потом разделить handlers и улучшать persistence только там, где этого действительно требует модель запуска бота.
